# Research: Implement Core ContinuousOrchestrator with APScheduler

**Date**: 2026-05-06

## 1. Executive Summary
This research maps the current state of task scheduling and worker execution in Omni Core V2. Currently, `main.py` acts as the primary orchestrator, initializing a global `BackgroundScheduler` and manually registering jobs that call `worker_manager.run_cycle`. The goal of the upcoming implementation is to encapsulate this logic within `WorkerManager` or a dedicated `ContinuousOrchestrator` to decouple background tasks from the UI/FastAPI entry point.

## 2. Technical Context
- **Affected Components**: `main.py`, `worker_manager.py`, `core/worker_base.py`.
- **Current Behavior**:
    - `main.py` (L176) initializes `scheduler = BackgroundScheduler()`.
    - Multiple jobs are registered via `scheduler.add_job` between L178 and L280 in `main.py`.
    - Most jobs use `lambda` functions to wrap calls like `worker_manager.run_cycle("WorkerName")`.
    - Special integration exists for `WeatherWorker` (L213-225) which updates a global `CURRENT_MOOD` variable.
- **Data Flow**:
    1. `main.py` starts -> `scheduler.start()` is called.
    2. Jobs trigger based on `IntervalTrigger` or `CronTrigger`.
    3. `worker_manager.run_cycle(name)` is invoked.
    4. `WorkerBase.execute_cycle()` (in `core/worker_base.py:75`) handles logging, execution, and persistence to `RewardStore`.

## 3. Findings & Analysis
- **APScheduler Patterns**: The project uses `BackgroundScheduler`, `IntervalTrigger`, and `CronTrigger` from `apscheduler` package (`main.py:105-107`).
- **Worker Manager Usage**: `worker_manager` is already imported and used in `main.py`. It provides a centralized way to trigger workers, but doesn't yet own the scheduler.
- **Job IDs**: Jobs in `main.py` use explicit IDs (e.g., `guardian_watchdog`, `curadoria_job`, `weather_sync`), which is a good pattern for management/replacement.
- **Misfire Grace Time**: `daily_playlist_gen` uses `misfire_grace_time=3600` (`main.py:291`), indicating concern for execution reliability.

## 4. Technical Constraints
- **Circular Imports**: Moving scheduling to `worker_manager.py` might trigger circular imports if workers (which depend on `core`) also reference the manager. (Currently `worker_manager.py` imports specific workers).
- **Global State**: `CURRENT_MOOD` is a global variable in `main.py` (L213) updated by `update_weather_mood_job`. Moving this job to an orchestrator requires a shared state mechanism (like `core/state.py`).
- **Singleton Pattern**: The `worker_manager_instance` in `worker_manager.py:47` is already used globally, which simplifies the transition.

## 5. Architecture Documentation
- **Worker Pattern**: All workers inherit from `WorkerBase` and implement `run_cycle`.
- **Execution Lifecycle**: `execute_cycle` (Base) -> `run_cycle` (Implementation) -> `record` (RewardStore).
- **Scheduling Philosophy**: Currently "Event-loop in Main". The goal is "Orchestrated by Manager".
