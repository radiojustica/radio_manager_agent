# Implement Core ContinuousOrchestrator with APScheduler - Implementation Plan

## Overview
This plan details the migration of task scheduling logic from `main.py` to `worker_manager.py`. By centralizing orchestration, we enable the modular backend to run autonomously and provide a cleaner interface for health monitoring.

## Scope Definition (CRITICAL)
### In Scope
- Refactor `core/state.py` to include `CURRENT_MOOD`.
- Enhance `WorkerManager` to initialize and manage a `BackgroundScheduler`.
- Implement `WorkerManager.start_orchestrator()` and `WorkerManager.stop_orchestrator()`.
- Migrate existing jobs from `main.py` to the new orchestrator.
- Add a mechanism to map workers to schedules.

### Out of Scope (DO NOT TOUCH)
- Creating new API endpoints (Ticket `f7c3e8d2`).
- Loading intervals from `settings.json` (Ticket `a1b2c3d4`).
- Enhancing `RewardStore` history (Ticket `e5f6g7h8`).

## Current State Analysis
- `main.py:176` initializes the global scheduler.
- `main.py:178-280` registers jobs manually.
- `worker_manager.py` is a passive registry of workers without execution logic.
- `core/state.py` is underutilized for cross-module state.

## Implementation Phases

### Phase 1: Shared State Refactoring
- **Goal**: Decouple `CURRENT_MOOD` from `main.py`.
- **Steps**:
  1. [x] Add `CURRENT_MOOD = "Ensolarado"` to `core/state.py`.
  2. [x] Update all references in `main.py` and workers to use `core.state.CURRENT_MOOD`.
- **Verification**: Run `main.py` and ensure the dashboard still displays the correct mood.

### Phase 2: Orchestrator Integration in WorkerManager
- **Goal**: Give `WorkerManager` the ability to schedule jobs.
- **Steps**:
  1. [x] Import `BackgroundScheduler` and triggers in `worker_manager.py`.
  2. [x] Add `self.scheduler = BackgroundScheduler()` to `WorkerManager.__init__`.
  3. [x] Implement `start_orchestrator()`: Starts the scheduler and registers default worker jobs.
  4. [x] Define a `DEFAULT_SCHEDULES` mapping in `worker_manager.py`:
     - `GuardianWorker`: 30s interval + 2s high-frequency.
     - `CuradoriaWorker`: 5m interval.
     - `WeatherWorker`: 30m interval (wrapped to update `state.CURRENT_MOOD`).
     - `SyncWorker`: 4h interval.
     - `AuditWorker`: 1h interval.
     - `PlaylistWorker`: Cron at 00:00.
- **Verification**: Instantiate `WorkerManager` in a script and verify `scheduler.get_jobs()` contains the expected entries.

### Phase 3: Migration and Cleanup
- **Goal**: Switch `main.py` to use the new orchestrator.
- **Steps**:
  1. [x] Remove manual `scheduler.add_job` calls from `main.py`.
  2. [x] Replace with `worker_manager_instance.start_orchestrator()`.
  3. [x] Ensure proper cleanup: Call `stop_orchestrator()` on application exit.
- **Verification**: Full system test. All workers should trigger as expected in logs.

### Phase 4: Error Handling & Resilience
- **Goal**: Ensure the orchestrator doesn't crash on worker failure.
- **Steps**:
  1. [x] Wrap job execution in a helper method that logs exceptions but keeps the scheduler alive. (Note: `WorkerBase.execute_cycle` already has a try/except, but the job wrapper should be equally safe).
- **Verification**: Force a worker to raise an exception and verify the scheduler continues running other jobs.
