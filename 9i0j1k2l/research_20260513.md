## 1. Estado Atual: `scripts/bulletin_sync.py`
O script atual é um mecanismo de espelhamento local que opera entre dois diretórios:
- **Origem (SOURCE):** `D:\SERVIDOR\DRIVE\RADIO TJRN CONTEÚDO\EDIÇÃO\BOLETINS` (Pasta espelhada pelo Google Drive Desktop).
- **Destino (TARGET):** `D:\SERVIDOR\BOLETINS` (Pasta de reprodução da rádio).

### Problemas Identificados:
1. **Dependência de Terceiros:** Depende totalmente do binário do Google Drive Desktop estar rodando e sincronizando. Se o app do Drive travar, o sistema "acha" que não há novos arquivos.
2. **Execução Manual/Ad-hoc:** Não está integrado ao ciclo de vida dos workers do `worker_manager.py`. É apenas um script solto.
3. **Lógica de Parsing Frágil:** O parsing via Regex (`(\d{2})_(\d{2})_(\d{4})_B(\d+)`) funciona, mas não há fallback para API oficial em caso de falha no espelhamento local.
4. **Falta de Monitoramento:** Não há registro de recompensas (Reward System) ou telemetria integrada ao OmniCore.

## 2. Proposta: `BulletinWorker`
Devemos transformar essa lógica em um Worker oficial para que o orquestrador gerencie a frequência e reporte falhas.

### Estrutura do Worker:
- **Classe:** `BulletinWorker(WorkerBase)`
- **Responsabilidade:** Chamar a lógica de sincronia e validar se os arquivos chegaram ao destino.
- **Integração:** Registrar no `worker_manager.py` para rodar em intervalos (ex: a cada 15 minutos).

## 3. Estratégia de API vs Local
- **Preferência:** API do Google Drive (PyDrive2 ou Google API Client).
- **Realidade:** Sem credenciais (`client_secrets.json`), o sistema continuará usando o espelhamento local (`D:`).
- **Hibridismo:** O `BulletinWorker` deve tentar o espelhamento local primeiro, mas deixar a porta aberta para uma `GoogleDriveService` no futuro.
