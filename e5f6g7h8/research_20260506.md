# Research: Implement Worker Execution History in RewardStore

**Date**: 2026-05-06

## 1. Executive Summary
This research investigates the `RewardStore` class and its current handling of worker telemetry and history. It finds that while a global history of the last 1000 execution cycles is maintained, there is no persistent per-worker rolling log. Filtering by worker currently requires scanning the entire global history list at runtime.

## 2. Technical Context
- **Data Structure**: `RewardStore` (`core/reward.py`) manages a `self.data` dictionary with two main keys:
    - `"workers"`: A map where each key is a worker name, containing `score_total`, `cycles`, and `last_result`.
    - `"history"`: A flat list of the last 1000 `record` dictionaries.
- **Recording Mechanism**: `record()` (`core/reward.py:46`) creates a `record` object, updates the worker's summary data, appends the record to the global history, and prunes the global history if it exceeds 1000 entries.
- **Retrieval Mechanism**: `history(worker_name, limit)` (`core/reward.py:105`) filters the global `self.data["history"]` by `worker_name` and returns the last `limit` entries. This is an O(N) operation where N is the size of the global history.

## 3. Findings & Analysis
- **Per-Worker History Lack**: Each worker entry in `self.data["workers"]` only stores the `last_result`. There is no dedicated history list within the worker object itself.
- **Global History Bloat**: The global history is capped at 1000 entries, but it mixes all workers. If many workers run frequently (like `GuardianWorker` every 2s), the history of slower workers (like `WeatherWorker` every 30m) might be quickly rotated out of the global list.
- **Data Flow**: `WorkerBase.execute_cycle()` calls `reward_store.record()` after each run, providing `score`, `violations`, and `metadata`.

## 4. Technical Constraints
- **JSON Size**: Adding per-worker history will increase the size of `worker_rewards.json`. Caps must be strictly enforced (as suggested in the ticket: 10-20 entries).
- **Persistence**: The implementation must ensure that per-worker history is saved and loaded correctly from the JSON file.

## 5. Architecture Documentation
- **Pattern**: `RewardStore` uses `json.dump` with `indent=4` for persistence.
- **Method Pattern**: Existing methods like `latest()` and `summary()` provide quick access to the `workers` sub-dictionary.
