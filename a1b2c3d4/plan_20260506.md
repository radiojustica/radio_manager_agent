# Implement Dynamic Scheduling and Fault Tolerance Implementation Plan

## Overview
Move worker scheduling intervals from hardcoded values in code to `settings.json` and enhance `WorkerManager` to handle and record manager-level exceptions in the `RewardStore`.

## Scope Definition (CRITICAL)
### In Scope
- Add `workers` section to `config/settings.json`.
- Update `WorkerManager` to load configuration and use it for scheduling.
- Refactor `WorkerManager.run_cycle` for improved fault tolerance.
- Record "CRITICAL_FAILURE" in `RewardStore` on unhandled exceptions.
### Out of Scope (DO NOT TOUCH)
- Changing `WorkerBase` or individual worker logic.
- Implementing a real-time config watcher (reload will happen on startup).
- Modifying `RewardStore` schema (use existing `record` method).

## Current State Analysis
- Intervals are hardcoded in `worker_manager.py:start_orchestrator()` [L64-120].
- `run_cycle` catches exceptions but only returns an error dict without persistent recording [L34-48].
- Configuration loading pattern exists in `director/grade_rules.py` but isn't used by `WorkerManager`.

## Implementation Phases

### Phase 1: Configuration & Loading
- **Goal**: Define the configuration structure and implement loading logic in `WorkerManager`.
- **Steps**:
  1. [x] Update `config/settings.json` with a `workers` section containing intervals for all registered workers.
  2. [x] Add `_load_config()` method to `WorkerManager` (modeled after `grade_rules.py`).
  3. [x] Call `_load_config()` in `__init__`.
- **Verification**: Print `self.config` in `__init__` or use a debugger to verify loaded values.

### Phase 2: Dynamic Orchestration
- **Goal**: Replace hardcoded intervals with values from configuration.
- **Steps**:
  1. [x] Refactor `WorkerManager.start_orchestrator()` to read intervals from `self.config`.
  2. [x] Ensure fallback to hardcoded defaults if a worker is missing from `settings.json`.
- **Verification**: Check logs to confirm workers are scheduled with expected intervals.

### Phase 3: Fault Tolerance & Persistent Error Tracking
- **Goal**: Ensure all execution failures are recorded in the `RewardStore`.
- **Steps**:
  1. [x] Refactor `WorkerManager.run_cycle`'s `except` block.
  2. [x] Inside `except`, call `self.reward_store.record()` with `score=-10`, `violations=["CRITICAL_MANAGER_FAILURE"]`, and metadata containing the exception message.
- **Verification**: 
  - Call `run_cycle` with a non-existent worker name via API or script.
  - Verify `worker_rewards.json` contains the failure record.
