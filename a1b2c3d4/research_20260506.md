# Research: Implement Dynamic Scheduling and Fault Tolerance

**Date**: 2026-05-06

## 1. Executive Summary
This research maps the current scheduling mechanism in `WorkerManager` and the configuration loading patterns in the project. It identifies that scheduling is currently hardcoded and that `WorkerManager.run_cycle` provides basic fault tolerance but lacks integration with the `RewardStore` for unhandled manager-level exceptions.

## 2. Technical Context
- **Current Scheduling**: Managed in `worker_manager.py:start_orchestrator()` using `APScheduler`. Intervals are hardcoded in the method (e.g., `IntervalTrigger(seconds=30)` for `GuardianWorker`, `IntervalTrigger(minutes=5)` for `CuradoriaWorker`).
- **Configuration Loading**: The pattern observed in `director/grade_rules.py:_carregar_config()` uses `config/settings.json` with a dictionary merge against hardcoded defaults.
- **Fault Tolerance**:
    - `WorkerBase.execute_cycle()` (`core/worker_base.py:73`) has a `try/except` block that catches worker-specific exceptions, records a -5 score in `RewardStore`, and returns a standard `WorkerResult`.
    - `WorkerManager.run_cycle()` (`worker_manager.py:34`) wraps the call to `execute_cycle()` in another `try/except`. If an error occurs here (e.g., worker not found or manager-level failure), it logs to the standard logger but **does not** record anything in the `RewardStore`.
- **Reward Tracking**: `RewardStore.record()` (`core/reward.py:46`) accepts a `worker_name`, `score` (int), `violations` (list), and `metadata` (dict). It maintains a persistent JSON file and an in-memory history of the last 1000 entries.

## 3. Findings & Analysis
- **Dynamic Scheduling Gap**: `settings.json` lacks a `workers` or `scheduling` section. `WorkerManager` does not currently read from any configuration file for its intervals.
- **Logging Inconsistency**: Unhandled exceptions in `WorkerManager.run_cycle` are "swallowed" and logged to `logging`, while internal worker errors are properly tracked in `RewardStore`.
- **Fault Tolerance Logic**: The nested `try/except` structure is good for isolation, but the outer layer needs to communicate failures to the persistent reward system to ensure "CRITICAL_FAILURE" states are visible to the station owner.

## 4. Technical Constraints
- **APScheduler Lifecycle**: Changes to scheduling usually require stopping and restarting the scheduler or dynamically removing/adding jobs.
- **JSON Structure**: Adding a new top-level key to `settings.json` is safe and follows project conventions.

## 5. Architecture Documentation
- **Configuration Pattern**: Merging `defaults` with `data.get("section", {})` from `settings.json`.
- **Worker Execution Flow**: `run_cycle` (Manager) -> `execute_cycle` (Base) -> `run_cycle` (Implementation).
