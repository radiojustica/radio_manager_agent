# Research: Implement Worker Status and Control API Endpoints

**Date**: 2026-05-06

## 1. Executive Summary
This research investigates the existing FastAPI router structure and the `WorkerManager` architecture to determine the best way to expose worker telemetry and control via API. The goal is to provide the frontend with visibility into background tasks and the ability to trigger them manually.

## 2. Technical Context
- **Affected Components**: `routers/workers.py` (to be created), `main.py`, `worker_manager.py`.
- **Current Behavior**:
    - `worker_manager.py` contains the `WorkerManager` class and a global instance `worker_manager_instance`.
    - `WorkerManager` maintains a dictionary of workers in `self.workers`.
    - `WorkerBase.health()` (in `core/worker_base.py:130`) returns a dictionary with name, running status, last run timestamp, and total reward score.
    - `WorkerManager.run_cycle(name)` (in `worker_manager.py:34`) triggers a specific worker's execution cycle and returns a standardized result.
- **Related Routers**:
    - `routers/status.py` handles general system telemetry (BUTT instances, ZaraRadio status, Now Playing).
    - `routers/acervo.py`, `routers/config.py`, etc., handle specific domain logic.

## 3. Findings & Analysis
- **API Pattern**: The project uses `APIRouter` with prefixes (e.g., `prefix="/api/status"` in `routers/status.py:13`).
- **Dependency Injection**: Routers use `Depends(get_db)` for database access, though worker management primarily interacts with the `worker_manager_instance`.
- **Worker Enumeration**: `worker_manager_instance.workers.values()` can be iterated to collect health data from all workers.
- **Control Mechanism**: `worker_manager_instance.run_cycle(name)` is the designated method for triggering a worker cycle.

## 4. Technical Constraints
- **Asynchronous Execution**: `run_cycle` is synchronous (blocking the thread during execution). Since it's triggered via API, it should ideally be run in a background thread or made async if it involves long-running operations. However, existing patterns in `routers/status.py` (like `sync_bulletins`) seem to run directly or use external threading.
- **Router Registration**: New routers must be included in `main.py` using `app.include_router()`.

## 5. Architecture Documentation
- **Worker Registry**: `WorkerManager` is the single source of truth for all modular background tasks.
- **Endpoint Structure**:
    - `GET /api/workers/status` -> Iterate `workers`, return list of `health()`.
    - `POST /api/workers/{name}/run` -> Call `run_cycle(name)`, return result.
